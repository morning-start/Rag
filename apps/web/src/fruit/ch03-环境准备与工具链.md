# 第3章 环境准备与工具链

前两章我们建立了对 RAG 的宏观认知：理解了大模型的局限、RAG 的核心价值，以及从 Naive RAG 到 Agentic RAG 的五代架构演进。从本章开始，我们正式进入动手实践阶段。本章将帮你完成两件事：选对工具、搭好环境。

工具选型决定了后续开发的效率和上限。选对了，事半功倍；选错了，后期重构成本极高。我们将采用**混合方案**——底层组件（文档加载、分块、嵌入、向量存储、LLM 调用）全部使用原生 SDK，让你深入理解每个环节的工作原理；编排层使用 LangGraph StateGraph 处理复杂的有向图流程（条件分支、循环、状态管理），避免重复造轮子。

---

## 3.1 技术选型概览

### 3.1.1 实现路线对比：框架 vs 原生 SDK vs 混合方案

构建 RAG 系统有三条路线可选，它们并非竞争关系，而是适用于不同场景。

**LangChain / LangGraph** 是当前最主流的通用型 LLM 应用开发框架，GitHub 星数超过 11.9 万。它的核心哲学是"模块化积木桶"——把 LLM 应用的所有环节都拆成标准化、可替换的组件。2026 年 2 月发布的 LangGraph 2.0 已成为企业级 Agent 编排的首选。但 LangChain 的抽象层较厚，调试困难，且版本迭代频繁导致 API 不稳定。

> **LangChain 的"抽象税"问题**：LangChain 将文档加载、分块、嵌入、检索、生成等环节封装为高度抽象的接口（如 `Document`、`VectorStore`、`BaseRetriever`），虽然降低了入门门槛，但也隐藏了大量实现细节。当你需要调试嵌入维度不匹配、分块边界截断语义、检索排序偏差等问题时，这些抽象层反而成为障碍。此外，LangChain 的版本迭代非常频繁（几乎每周都有 breaking change），生产环境中经常遇到升级后代码无法运行的情况。因此，本教程**不使用 LangChain 的封装层**，但会在编排层使用其子项目 LangGraph——因为编排层解决的问题（状态管理、条件分支、循环执行）确实需要专门的框架来处理。

**LlamaIndex** 是 RAG 领域的垂直标杆，GitHub 星数超过 4.4 万。它的设计哲学是"数据优先"——从数据摄入、分块、索引到检索、查询引擎，所有能力都围绕"如何让 LLM 更好地理解你的私有数据"做了深度优化。在纯 RAG 场景下检索效果通常优于 LangChain。

> **LlamaIndex 的核心能力**：LlamaIndex 围绕"数据连接 → 索引构建 → 查询检索 → 回答合成"这一管线做了深度抽象，其核心概念包括：
> - **Index Structures（索引结构）**：提供 VectorStoreIndex（向量索引）、KeywordTableIndex（关键词表索引）、KnowledgeGraphIndex（知识图谱索引）等多种索引类型，可单独使用也可组合为复合索引。每种索引都针对特定的检索模式做了优化。
> - **Query Engine（查询引擎）**：封装了"接收查询 → 检索 → 合成回答"的完整流程，支持子问题查询（SubQuestionQueryEngine）、SQL 查询（SQLQueryEngine）等多种查询策略。
> - **Response Synthesizer（回答合成器）**：负责将检索到的上下文片段组装成最终回答，支持 refine（逐步精炼）、compact（压缩合并）、tree_summarize（树状摘要）等多种合成模式。
> - **Data Connectors（数据连接器）**：内置 160+ 种数据源连接器（PDF、Notion、Slack、数据库等），开箱即用。
>
> LlamaIndex 的优势在于 RAG 场景的深度优化，但在 Agent 编排和多步推理方面不如 LangGraph 灵活。本教程选择不使用 LlamaIndex，是为了让读者从原生 SDK 层面理解 RAG 的每个环节。
>
> **LlamaIndex 最小使用示例**：
>
> ```python
> from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
> from llama_index.llms.ollama import Ollama
>
> # 配置本地模型
> Settings.llm = Ollama(model="llama3.3:8b", request_timeout=120)
>
> # 加载文档并构建索引（一行搞定）
> documents = SimpleDirectoryReader("./data").load_data()
> index = VectorStoreIndex.from_documents(documents)
>
> # 创建查询引擎并提问
> query_engine = index.as_query_engine(similarity_top_k=5)
> response = query_engine.query("RAG 的核心优势是什么？")
> print(response)
> ```

**原生实现**即直接调用各组件的原生 SDK，不依赖任何框架。优点是完全可控、零抽象开销、代码透明；缺点是所有组件需要自己拼装，适合深度定制或学习场景。

**混合方案**（本教程选择）结合了原生实现和框架的优势：底层组件使用原生 SDK，编排层使用 LangGraph StateGraph。

| 对比维度 | LangChain/LangGraph | LlamaIndex | 原生实现 | **混合方案（本教程）** |
|----------|-------------------|------------|---------|---------------------|
| 核心赛道 | 通用型 LLM 应用编排 | RAG 与检索增强专项 | 完全自定义 | 原生组件 + 图编排 |
| GitHub 星数 | 119K+ | 44K+ | — | — |
| 学习曲线 | 较陡（抽象层多） | 较缓（数据聚焦） | 中等（需了解各组件） | 中等 |
| 调试难度 | 高（黑盒多） | 中 | 低（完全透明） | 低（底层透明） |
| 生产部署 | LangSmith 可观测性 | LlamaCloud 托管 | 自建 | 自建 + LangGraph |
| 适用场景 | 复杂工作流 + Agent | 文档密集型 RAG | 学习 / 深度定制 | **学习 + 复杂编排** |

（来源: [06-RAG全栈无LangChain技术选型.md](reference/04-工具链与环境/06-RAG全栈无LangChain技术选型/)）

**为什么选择混合方案？**

本教程的技术选型策略如下：

| 层级 | 技术选择 | 理由 |
|------|---------|------|
| 文档加载 | PyMuPDF / python-docx（原生 SDK） | 直接操作文档对象模型，理解解析细节 |
| 文本分块 | 自定义函数 | 掌握分块策略的核心逻辑（chunk size、overlap、语义边界） |
| 向量嵌入 | SentenceTransformer（原生 SDK） | 理解嵌入模型如何将文本映射到向量空间 |
| 向量存储 | ChromaDB 原生 SDK | 掌握向量数据库的增删改查和相似度检索原理 |
| LLM 调用 | httpx + Ollama REST API | 理解 LLM API 的请求/响应协议，支持异步 |
| **编排层** | **LangGraph StateGraph** | **处理 Agentic RAG 中的条件分支、循环、状态管理** |

这个策略的核心思想是：**底层组件用原生 SDK，让你"知其然，知其所以然"；编排层用 LangGraph，让你不必重复造轮子**。

为什么编排层需要框架？因为 Agentic RAG（第 5 章核心实战）涉及复杂的流程控制：检索结果不够时要重新检索、需要调用工具获取外部信息、需要根据不同条件走不同分支、需要维护多轮对话的状态。这些需求本质上是有向图（DAG）的执行问题，而 LangGraph StateGraph 正是为解决这类问题而设计的——它提供了类型安全的状态管理、条件边（conditional edges）、循环检测、检查点恢复等能力。手动实现这些功能不仅代码量大，而且容易引入 bug。

为什么不直接用 LangChain 的全套封装？因为 LangChain 的封装层（如 `ChatOllama`、`Chroma.from_documents()`、`RecursiveCharacterTextSplitter`）会隐藏底层细节，让你无法真正理解 RAG 的每个环节如何工作。当你遇到嵌入维度不匹配、分块边界截断语义、检索排序偏差等问题时，这些抽象层反而成为调试障碍。

### 3.1.2 大模型选择

RAG 系统中的大模型负责"理解检索结果并生成回答"。模型选择需要在质量、成本、延迟、隐私四个维度之间做权衡。

**云端 API 模型**（OpenAI、Anthropic、Google）的优势是效果最好、无需运维，但存在数据外传风险和持续的 API 成本。OpenAI 的 GPT-4o 和 text-embedding-3-small/large 是目前最成熟的商用组合。

**本地部署模型**通过 Ollama 等工具在本地运行开源模型，数据完全不出机房，适合对隐私敏感的企业场景。2025 年后 Ollama 已支持超过 1700 个模型，包括 Llama 3.3、DeepSeek-R1、Qwen 2.5、Gemma 3 等主流开源模型。最低 8GB 内存即可运行 7B 参数级别的模型，16GB 以上可流畅运行 13B 模型。

**HuggingFace 开源模型**提供最大的灵活性，你可以从社区下载任意模型并自由微调，但需要自行处理部署和推理优化。

| 选择维度 | 云端 API | 本地部署（Ollama） | HuggingFace 自部署 |
|---------|---------|-------------------|-------------------|
| 数据隐私 | 数据外传 | 完全本地 | 完全本地 |
| 单次调用成本 | $0.01-$0.06/1K token | 免费（电费除外） | 免费（电费除外） |
| 初始搭建成本 | 零 | 需 GPU 硬件 | 需 GPU 硬件 |
| 延迟 | 网络依赖 200-800ms | 本地推理 50-300ms | 取决于硬件 |
| 模型效果 | GPT-4o/Claude 3.5 最优 | Llama 3.3/Qwen 2.5 接近商用 | 取决于模型 |

**RAG 推荐搭配**：

| 场景 | Embedding 模型 | 生成模型 | 说明 |
|------|---------------|---------|------|
| 快速原型 | text-embedding-3-small | GPT-3.5-Turbo | 成本最低，效果够用 |
| 高质量生产 | text-embedding-3-large | GPT-4o | 效果最优 |
| 隐私优先 | BGE-M3 / M3E | Llama 3.3 / Qwen 2.5 | 全链路本地 |
| 多语言场景 | BGE-M3 | Qwen 2.5 | 中英双语优化 |
| 预算有限 | nomic-embed-text（Ollama） | Llama 3.3 8B（Ollama） | 零 API 成本 |

（来源: RAG System Design Complete Guide, GitHub; MarsDevs Framework Guide, 2026）

### 3.1.3 向量数据库对比

向量数据库是 RAG 系统的"记忆仓库"，负责存储文档的向量表示并执行相似度检索。本教程将使用 **ChromaDB 原生 SDK** 进行操作，不依赖 LangChain 封装层。（来源: [02-ChromaDB原生SDK使用指南.md](reference/05-数据处理与索引/02-ChromaDB原生SDK使用指南/)）

**Chroma** 是最适合快速上手的选择。极简 API，5 行代码即可完成向量存储和检索。缺点是不适合大规模生产环境（建议 10 万向量以下）。

**FAISS**（Facebook AI Similarity Search）是 Meta 开源的向量检索库，性能极致，支持 20+ 种索引算法和 GPU 加速。但它不是完整的数据库——没有持久化服务、没有分布式支持、没有 REST API。

> **FAISS 的前世今生**：FAISS 诞生于 2017 年的 Facebook AI Research（FAIR）实验室，最初是为了解决十亿级向量的快速近邻搜索问题。在 FAISS 出现之前，精确的 KNN 搜索在大规模数据集上的时间复杂度为 O(n*d)，根本无法满足生产需求。FAISS 引入了倒排文件索引（IVF）、乘积量化（PQ）、HNSW 等近似最近邻（ANN）算法，将搜索复杂度降低到亚线性级别。2024 年 FAISS 已支持 GPU 加速和 20+ 种索引类型，成为学术界和工业界的事实标准。
>
> **FAISS 最小使用示例**：
>
> ```python
> import faiss
> import numpy as np
>
> # 生成示例数据：10000 个 128 维向量
> dimension = 128
> np.random.seed(42)
> vectors = np.random.random((10000, dimension)).astype("float32")
>
> # 构建 IndexFlatL2 索引（精确搜索，适合小数据集）
> index = faiss.IndexFlatL2(dimension)
> index.add(vectors)
> print(f"索引中向量数量: {index.ntotal}")  # 10000
>
> # 检索与查询向量最相似的 Top-5
> query = np.random.random((1, dimension)).astype("float32")
> distances, indices = index.search(query, k=5)
> print(f"最近邻索引: {indices[0]}")
> print(f"对应距离: {distances[0]}")
> ```
>
> **FAISS 使用注意事项**：
> - FAISS 是一个**库**而非数据库，不提供持久化存储服务——你需要自己管理向量的保存和加载（通过 `faiss.write_index()` / `faiss.read_index()`）。
> - `IndexFlatL2` 是暴力精确搜索，适合万级以下数据；百万级数据应使用 `IndexIVFFlat` 或 `IndexHNSWFlat` 等近似索引。
> - FAISS 只接受 `float32` 类型的 numpy 数组，输入前务必检查数据类型。
> - FAISS 不支持元数据过滤——如果你需要"按文档类型过滤后检索"，需要自行实现或使用 ChromaDB/Qdrant。

**Milvus** 是生产环境的首选。分布式架构支持十亿级向量，多种索引算法（HNSW、IVF、GPU 加速），云原生设计。百万级向量 QPS 可达 1000+，检索延迟低于 100ms。

**Qdrant** 用 Rust 编写，性能优秀，原生支持混合检索（稀疏+密集向量），过滤条件查询的 P95 延迟仅 22ms。

| 数据库 | 类型 | 建议规模 | 部署难度 | 混合检索 | 适用场景 |
|-------|------|---------|---------|---------|---------|
| Chroma | 开源轻量 | <10 万向量 | ⭐ | 需手写 | 快速原型、本地开发 |
| FAISS | 库（非 DB） | 取决于内存 | ⭐⭐ | 需手写 | 离线批处理、算法研究 |
| Qdrant | 开源高性能 | 百万-十亿 | ⭐⭐⭐ | 原生支持 | 生产环境、复杂过滤 |
| Milvus | 开源分布式 | 十亿-千亿 | ⭐⭐⭐⭐ | 原生支持 | 大规模企业应用 |
| Pinecone | 商业托管 | 百万-百亿 | ⭐ | 原生支持 | 快速上线、零运维 |

（来源: 向量数据库选型指南, GitHub AgentGuide; CSDN RAG 向量数据库对比, 2025）

### 3.2.2 核心依赖安装

RAG 系统的核心依赖可分为四层：文档处理层、Embedding 层、向量存储层和模型调用层。底层组件使用原生 SDK，编排层使用 LangGraph。

**一次性安装全部核心依赖**：

```bash
pip install chromadb pymupdf python-docx \
    sentence-transformers httpx \
    numpy tqdm langgraph
```

**各依赖用途说明**：

| 依赖包 | 用途 | 必要性 |
|-------|------|-------|
| chromadb | 向量数据库（原生 SDK） | 核心 |
| pymupdf | PDF 文档解析（替代 PyPDFLoader） | 文档处理时必选 |
| python-docx | Word 文档解析（替代 Docx2txtLoader） | Word 文档时必选 |
| sentence-transformers | 开源 Embedding 模型（替代 OllamaEmbeddings） | 本地 Embedding 时必选 |
| httpx | HTTP 请求（用于调用 Ollama/OpenAI API） | 调用 LLM 时必选 |
| numpy | 数组运算（向量处理） | 核心 |
| tqdm | 进度条显示 | 推荐 |
| langgraph | Agent 编排层（StateGraph） | 第 5 章实战时必选 |

> **为什么选择 httpx 而不是 requests？**
>
> `requests` 是 Python HTTP 库的经典选择，但它诞生于 2011 年，存在以下局限：不支持异步（`requests` 只有同步 API）、底层使用 urllib3 且维护节奏放缓、不支持 HTTP/2。
>
> `httpx` 是更现代的替代方案，它提供了与 `requests` 几乎一致的 API（降低迁移成本），同时支持同步和异步两种调用模式。在 RAG 系统中，异步能力非常重要——当你需要并发调用多个 LLM 请求、同时检索多个向量库、或并行处理多个文档时，`httpx.AsyncClient` 可以显著提升吞吐量。此外，`httpx` 原生支持 HTTP/2，在连接复用和性能上优于 `requests`。本教程从第 5 章开始会大量使用异步调用，因此统一使用 `httpx`。

### 3.2.3 Ollama 本地部署与模型拉取

Ollama 是本地部署大模型的最简方案，它将模型权重、配置和运行环境打包成一个整体，让本地部署变得像运行一个普通软件一样简单。

**安装 Ollama**

**Windows**（推荐方式）：

```bash
winget install Ollama.Ollama
```

或者从 https://ollama.com/download 下载安装程序，双击运行即可。

**macOS**：

```bash
brew install ollama
```

**Linux**：

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

**验证安装**：

```bash
ollama --version   # 应显示 ollama version 0.5.x 或更高
```

**拉取模型**

安装完成后，通过 `ollama pull` 命令下载模型。以下是 RAG 开发中推荐的模型组合：

**常用模型规格对比**：

**通过原生 REST API 调用模型**：

Ollama 默认在 `http://localhost:11434` 提供 OpenAI 兼容的 API 接口。我们使用 `httpx` 库直接调用，无需任何框架封装：

```python
import httpx

def call_ollama(prompt: str, model: str = "llama3.3:8b") -> str:
    """调用 Ollama 本地模型的原生 REST API（使用 httpx）"""
    response = httpx.post(
        "http://localhost:11434/api/generate",
        json={
            "model": model,
            "prompt": prompt,
            "stream": False
        },
        timeout=120.0  # 本地推理可能较慢，设置充足超时
    )
    result = response.json()
    return result["response"]

answer = call_ollama("用一句话解释什么是 RAG")
print(answer)
```

这段代码直接通过 HTTP POST 请求与 Ollama 通信，返回纯文本响应。完全透明，没有任何黑盒。使用 `httpx` 替代 `requests` 的好处是：API 风格几乎一致，但额外获得了异步支持（`httpx.AsyncClient`）和更现代的超时管理。后续第 5 章的 Agentic RAG 实战中，我们将利用 `httpx` 的异步能力并发调用多个 LLM 请求。

### 3.2.5 环境验证：最小可用 RAG（混合方案版）

完成上述步骤后，运行以下脚本验证整个环境是否就绪。这段代码构建了一个底层组件完全使用原生 SDK 的最小 RAG 流程：加载一段文本 → 分块 → 嵌入 → 存入 ChromaDB → 检索 → 生成回答。编排层的 LangGraph StateGraph 将在第 5 章引入。（来源: [02-ChromaDB原生SDK使用指南.md](reference/05-数据处理与索引/02-ChromaDB原生SDK使用指南/)）

```python
import os
import chromadb
from chromadb.utils import embedding_functions
import httpx

# ========== 1. 准备测试文本 ==========
text = """RAG（检索增强生成）是一种结合检索和生成的技术架构。
它通过从外部知识库中检索相关信息，将其作为上下文提供给大语言模型，
从而生成更准确、更有依据的回答。RAG 的核心优势在于：
能够利用最新的知识，减少幻觉，并支持私有数据的安全使用。"""

os.makedirs("data/sample", exist_ok=True)
with open("data/sample/test.txt", "w", encoding="utf-8") as f:
    f.write(text)

# ========== 2. 自定义文本分块函数 ==========
def split_text(text: str, chunk_size: int = 100, overlap: int = 20) -> list:
    """简单的文本分块函数，按字符切割并保留重叠"""
    chunks = []
    start = 0
    text_len = len(text)

    while start < text_len:
        end = min(start + chunk_size, text_len)
        chunks.append(text[start:end])
        start += (chunk_size - overlap) if end < text_len else chunk_size

    return chunks

chunks = split_text(text, chunk_size=100, overlap=20)
print(f"分块完成，共 {len(chunks)} 个文本块")

# ========== 3. 初始化 ChromaDB（原生 SDK）==========
client = chromadb.PersistentClient(path="./vectorstore/chroma_db")

embedding_func = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="all-MiniLM-L6-v2"
)

collection = client.get_or_create_collection(
    name="rag_documents",
    embedding_function=embedding_func,
    metadata={"hnsw:space": "cosine"}
)

# 清空旧数据（方便重复运行）
try:
    collection.delete(where={"source": "test.txt"})
except Exception:
    # 首次运行时集合中可能没有匹配数据，忽略即可
    pass

# 添加文档
collection.add(
    documents=chunks,
    metadatas=[{"source": "test.txt", "chunk_id": i} for i in range(len(chunks))],
    ids=[f"chunk_{i}" for i in range(len(chunks))]
)
print(f"已向 ChromaDB 添加 {len(chunks)} 个文档")

# ========== 4. 语义检索 ==========
query = "RAG 的核心优势是什么？"
results = collection.query(
    query_texts=[query],
    n_results=3
)

retrieved_docs = results['documents'][0]
print(f"检索到 {len(retrieved_docs)} 个相关文档")

# ========== 5. 构造 Prompt 并调用 LLM ==========
context = "\n\n".join(retrieved_docs)

prompt = f"""基于以下上下文回答问题。如果上下文中没有相关信息，请说明不知道。

上下文：
{context}

问题：{query}

请回答："""

# 通过 Ollama REST API 调用生成模型（使用 httpx）
response = httpx.post(
    "http://localhost:11434/api/generate",
    json={
        "model": "llama3.3:8b",
        "prompt": prompt,
        "stream": False
    },
    timeout=120.0
)
result = response.json()["response"]

print("\n" + "="*50)
print("RAG 回答：")
print(result)
print("="*50)
```

**代码解析**：

这个最小 RAG 示例展示了混合方案中底层组件的完整流程：

1. **分块**：自定义 `split_text()` 函数，按固定长度切割文本并保留重叠区域，替代了 LangChain 的 `RecursiveCharacterTextSplitter`
2. **嵌入与存储**：使用 ChromaDB 原生 SDK 的 `PersistentClient` + `SentenceTransformerEmbeddingFunction`，替代了 LangChain 的 `Chroma.from_documents()`
3. **检索**：直接调用 `collection.query()` 方法，传入查询文本和返回数量
4. **生成**：使用 `httpx.post()` 直接调用 Ollama 的 `/api/generate` 接口，替代了 LangChain 的 `ChatOllama`。相比 `requests`，`httpx` 提供了更现代的超时管理和异步支持
5. **Prompt 构建**：使用 Python f-string 格式化模板，替代了 LangChain 的 `ChatPromptTemplate`
6. **错误处理**：`collection.delete()` 使用 `try-except` 包裹，避免首次运行时因集合为空而抛出异常

每个底层环节都是透明的、可调试的、无隐藏逻辑的。这就是混合方案中"原生 SDK 层"的核心优势——你完全掌控每个组件的行为。而编排层的 LangGraph StateGraph 将在第 5 章引入，届时你会看到它如何优雅地管理复杂的多步推理流程。

如果这段代码能正常输出关于 RAG 核心优势的回答，说明你的开发环境已经完全就绪。接下来的第 4 章和第 5 章，我们将分别深入数据处理管线和检索生成管线的每一个环节。
