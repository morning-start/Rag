# 第7章 生成质量优化

上一章我们聚焦检索质量优化。本章进入**生成质量优化**——当检索到相关的上下文后，如何让 LLM 生成更准确、更有据可依、更少幻觉的回答。

生成质量是 RAG 系统的"最后一公里"。检索质量再高，如果 LLM 不能正确利用检索到的信息，或生成的回答存在幻觉，用户体验依然会很差。2026 年的数据显示：AI 幻觉在全球范围内造成的经济损失高达 **674 亿美元**，**47%** 的企业 AI 用户承认基于完全虚构的信息做出过重大商业决策。（来源: [AI响应溯源与幻觉抑制技术](reference/07-生成与提示工程/01-AI响应溯源与幻觉抑制技术.md)）

## 7.1 Prompt 工程深度实践

### 7.1.1 System Prompt 六要素设计

System Prompt 是 LLM 行为的"宪法"。生产级 RAG System Prompt 应包含六大核心要素：

| 要素 | 作用 | 缺失后果 |
|------|------|---------|
| 角色设定 | 定义 LLM 的身份和职责边界 | 行为不可预测 |
| 行为约束 | 规定哪些能做、哪些不能做 | 容易越权回答 |
| 格式规范 | 统一输出的风格和结构 | 输出格式混乱 |
| 引用溯源 | 强制标注来源 | 无法追溯验证 |
| 异常处理 | 规定边界情况下的行为 | 遇到异常时乱答 |
| 安全护栏 | 防止注入攻击和越权操作 | 存在安全风险 |

```python
SYSTEM_PROMPT = """
你是一个专业的知识库助手，专门为用户提供准确、有据可依的回答。

你的核心行为准则：
1. 严格基于提供的上下文信息回答问题
2. 如果上下文中没有相关信息，明确告知用户"根据提供的资料，我无法找到相关信息"
3. 回答中必须引用具体的来源文档编号，格式为 [来源X]
4. 不要编造、推测或补充上下文中不存在的信息
5. 如果上下文信息存在矛盾，指出矛盾并分别说明

你的回答应该：
- 结构清晰，分点论述
- 使用准确的专业术语
- 对复杂概念提供简要解释
"""

def build_rag_prompt(context: str, question: str) -> str:
    return f"""{SYSTEM_PROMPT}

上下文：{context}
问题：{question}

请回答："""
```

核心思想：**约束前置**。其中**引用溯源**和**异常处理**是 RAG 最关键的两个要素。（来源: [RAG全链路优化方案](reference/06-检索策略与优化/01-RAG全链路优化方案体系.md)）

### 7.1.2 Few-shot 示例增强

Few-shot 通过示例引导 LLM 学习正确的回答风格，特别适合解决"边界情况"：

```python
FEW_SHOT_PROMPT = """
你是一个专业的知识库助手。请参考以下示例来回答问题。

示例1：
上下文：RAG 通过从外部知识库检索相关信息来增强 LLM。
问题：什么是 RAG？
回答：RAG（Retrieval-Augmented Generation）是一种技术架构 [来源1]：
1. 从外部知识库检索与问题相关的信息
2. 将检索到的信息作为上下文提供给 LLM
3. LLM 基于上下文生成回答

示例2：
上下文：本文档介绍了 OAuth 2.0 授权框架。
问题：HTTP 状态码 500 表示什么？
回答：根据提供的上下文信息，我无法找到关于 HTTP 状态码 500 的信息。

上下文：{context}
问题：{question}
回答："""
```

生产环境建议维护 **Few-shot 示例池**，按场景分类，构建 Prompt 时动态选取最匹配的 2-3 个示例注入。示例需定期审核更新，保持与知识库同步。

### 7.1.3 CoT (Chain-of-Thought) for RAG：逐步推理提示 + 检索证据链

Chain-of-Thought（思维链）是一种引导大语言模型逐步推理后再给出最终答案的 Prompt 技术。将 CoT 引入 RAG 场景后，会产生独特的协同效应——RAG 提供事实基础，CoT 确保逻辑链条完整且可见。

#### 为什么 CoT 对 RAG 特别有效

传统 RAG 的生成过程是一个"黑盒"：检索到的上下文直接喂给 LLM，模型内部如何处理这些信息、是否真正使用了证据、推理过程是否合理——全部不可见。CoT 通过要求模型**展示推理过程**（show your work），带来三个关键优势：

1. **推理过程成为 checkpoint（检查点）**：每个中间步骤都可以独立验证其事实准确性
2. **Token 概率分布被中间步骤约束**：CoT 将生成过程从"预测最可能答案"转变为"构建推理链→推导答案"，减少随机错误
3. **错误定位更精确**：当最终答案出错时，可以通过回溯推理链快速定位是哪一步出了问题

（来源: [CoT-RAG幻觉抑制研究](reference/07-生成与提示工程/03-CoT-RAG幻觉抑制研究.md)）

#### Zero-shot CoT vs Few-shot CoT

| 形式 | 方法 | Token 开销 | 适用场景 | 效果提升 |
|------|------|-----------|----------|---------|
| **Zero-shot CoT** | 仅添加"请逐步思考"指令 | 低（+10-20%） | 快速启用、无标注数据 | 准确率 +15-25% |
| **Few-shot CoT** | 提供 2-3 个完整推理链示例 | 中（+30-50%） | 有标注数据、高质量要求 | 准确率 +20-35% |

对于 RAG 系统，推荐从 **Zero-shot CoT** 开始起步，在积累足够的优质输出样本后再升级到 Few-shot CoT。

#### ACL Findings 2025 关键发现

Texas A&M 大学的研究团队在 ACL Findings 2025 上发表了 CoT 与 RAG 联合使用的关键发现：

**正面效应**：
- CoT 在 HaluEval、FEVER、TruthfulQA 三个基准数据集上均显著降低幻觉频率
- Self-consistency（多次生成取一致性答案）进一步降低随机误差
- Self-verification（自我验证）引入事后验证步骤提升事实准确性

**重要警示**：
- **CoT 改变幻觉的可检测性**：降低错误率的同时，使残留幻觉更难被自动检测系统识别
- 推理链可能读起来逻辑完美但仍包含虚构事实——"精心修饰的错误答案比明显困惑的答案更难捕捉"
- 结构化推理链平滑了检测系统依赖的 erratic token probabilities（不稳定 token 概率）

（来源: [CoT-RAG幻觉抑制研究](reference/07-生成与提示工程/03-CoT-RAG幻觉抑制研究.md)）

**CoT 幻觉的"隐蔽性悖论"**：CoT 虽显著降低幻觉，但使残留幻觉更难被自动检测系统捕捉——Token Probability Audit 检出率从 78.3% 降至 41.2%，DetectGPT 从 72.1% 降至 38.6%。原因是 CoT 将生成过程分解为中间步骤后，token 概率分布变得更加平滑，削弱了检测工具依赖的统计特征。

**应对策略**：

1. **证据锚定验证**：强制每个推理步骤引用文档原文片段，自动化字符串匹配验证
2. **多路径交叉验证**：对同一问题生成 3-5 条独立 CoT 链，比对证据和结论
3. **步骤级独立核查**：将每个中间步骤独立提取，逐一验证事实性

```python
async def step_level_verify(cot_output: str, documents: str) -> list:
    """步骤级独立核查：将 CoT 推理链拆分为独立步骤逐一验证。"""
    import re
    steps = re.findall(r'步骤\d+\s*[-—]\s*(.+?)(?=步骤\d+\s*[-—]|=== 最终答案 ===)',
                       cot_output, re.DOTALL)
    results = []
    for i, step in enumerate(steps, 1):
        check = await call_ollama([
            {"role": "system", "content":
             "判断以下推理步骤是否完全基于给定文档。只回复 pass 或 fail。"},
            {"role": "user", "content":
             f"文档：{documents}\n\n推理步骤{i}：{step.strip()}"}
        ], temperature=0.0)
        results.append({"step": i, "passed": check.strip().lower().startswith("pass"),
                        "content": step.strip()[:100]})
    return results
```

#### 完整 CoT RAG Prompt 模板

以下是生产级 CoT RAG Prompt 模板，使用原生 Python + httpx 调用 Ollama：

```python
import httpx
import json

COT_RAG_SYSTEM = """你是一个严谨的知识库分析助手。
你的任务是基于检索到的文档，通过逐步推理得出结论。

核心规则：
1. 每个推理步骤必须引用具体的文档片段，格式为 [证据: 来源X]
2. 如果某一步推理缺乏文档支持，明确标注"该步骤无直接证据"
3. 最终结论必须是推理链条的逻辑结果，不得引入未提及的信息
4. 当文档信息不足以支撑完整推理时，在对应步骤注明"信息不足"
"""

def build_cot_rag_prompt(context: str, question: str) -> str:
    return f"""{COT_RAG_SYSTEM}

【检索文档】
{context}

【用户问题】
{question}

请严格按照以下格式输出：

=== 逐步推理 ===
步骤1 - 信息提取：从文档中提取与问题直接相关的关键信息点
[在此列出提取的关键点，每个点标注来源]

步骤2 - 关系分析：分析各信息点之间的逻辑关系和因果关系
[在此分析信息点之间的关联]

步骤3 - 推导结论：基于上述分析和文档内容，逐步推导出答案
[在此展示推导过程]

=== 最终答案 ===
[基于以上推理给出简洁明确的最终答案]

=== 证据来源汇总 ===
[列出所有引用的来源编号]
"""

async def call_llm_cot_rag(prompt: str, model: str = "qwen2.5:7b") -> str:
    async with httpx.AsyncClient(timeout=60.0) as client:
        response = await client.post(
            "http://localhost:11434/v1/chat/completions",
            json={
                "model": model,
                "messages": [{"role": "user", "content": prompt}],
                "temperature": 0.1,
                "max_tokens": 2048
            }
        )
        result = response.json()
        return result["choices"][0]["message"]["content"]
```

**技术解读**: 这个模板的设计遵循"先检索后推理、推理锚定证据"的原则。`=== 逐步推理 ===` 区块强制模型将思维过程外化，`[证据: 来源X]` 标注确保每一步都可追溯到原始文档。最后的 `=== 证据来源汇总 ===` 方便下游做自动化的事实核查。

#### 适用 / 不适用场景速查表

| 场景 | 推荐使用 | 避免使用 | 原因 |
|------|----------|----------|------|
| 多步数学 / 逻辑问题 | ✅ CoT | 直接生成 | 需要中间推理步骤 |
| 文档摘要需验证声明 | ✅ CoT + RAG | 单一方法 | 需要逐步核查 |
| 代码调试追踪逻辑 | ✅ CoT | 黑盒调用 | 需要可见推理路径 |
| 法律 / 医疗问答 | ✅ 全套方案 | 无防护 | 错误后果严重 |
| 简单事实查找 | | ❌ CoT | 增加不必要的延迟和成本 |
| 高吞吐批处理 (<2s) | | ❌ CoT/CoVe | 延迟预算严格 |
| 快速创意头脑风暴 | | ❌ CoT | 速度比精确性更重要 |

（来源: [CoT-RAG幻觉抑制研究](reference/07-生成与提示工程/03-CoT-RAG幻觉抑制研究.md)）

## 7.2 幻觉抑制策略

幻觉是 LLM 的根本性局限。在 RAG 场景中，幻觉表现为：生成的回答与检索到的上下文矛盾，或凭空编造上下文中不存在的信息。本章将从分类诊断、引用溯源、自反思管线、前沿方案和参数锁死五个层面系统性地解决幻觉问题。

### 7.2.1 幻觉的三种类型

理解幻觉的类型有助于针对性防御。综合学术界 2025 年最新分类法，幻觉可分为三大类别：

| 类别 | 典型表现 | 根本原因 | 主要解决策略 |
|------|----------|----------|--------------|
| **知识型幻觉** | 编造不存在的事实、引用虚假来源、过时信息 | 模型训练数据截止、领域知识缺失 | **RAG**（检索增强生成） |
| **逻辑型幻觉** | 数学计算错误、因果倒置、推理自相矛盾 | 多步推理能力不足、中间步骤出错 | **CoT**（思维链）、符号推理 |
| **输入冲突幻觉** | 与用户输入或检索上下文矛盾 | 注意力机制偏差、上下文干扰 | **引用溯源** + **Self-RAG** |

（来源: [LLM幻觉缓解综合综述2025](reference/07-生成与提示工程/05-LLM幻觉缓解综合综述2025.md)）

在 RAG 场景中，**上下文冲突幻觉**危害最大——用户提供了正确材料，模型却"视而不见"甚至与之矛盾。这也是本章重点解决的问题。

### 7.2.2 引用溯源（Citation Grounding）

引用溯源是降低幻觉最直接的方法——要求 LLM 在回答中标注每个论点的来源。2026 年的多项研究表明，强制引用溯源可以将幻觉率降低 **35%-50%**。（来源: [AI响应溯源与幻觉抑制技术](reference/07-生成与提示工程/01-AI响应溯源与幻觉抑制技术.md)）

```python
def build_citation_prompt(context: str, question: str) -> str:
    return f"""请基于以下上下文回答问题。
要求：
1. 回答中的每个关键论点必须标注来源，格式为 [来源X]
2. 如果某个论点综合了多个来源的信息，标注为 [来源X,Y]
3. 不要添加上下文中没有的信息
4. 如果上下文不足以回答某部分，明确说明"该部分信息不足"

上下文：{context}
问题：{question}

回答："""
```

引用溯源的效果高度依赖检索质量——再严格的引用格式也只是"精确地引用错误信息"，必须与第6章的检索质量优化配合使用。

### 7.2.2b 轻量级幻觉检测方案（无需 Self-RAG）

并非所有项目都需要引入 Self-RAG 这样完整的自反思管线。对于中小型项目或快速原型阶段，以下两个独立方案可以在不增加架构复杂度的前提下提供有效的幻觉防护。

#### 轻量级置信度评估

核心思路：利用 LLM 自身对生成结果进行置信度打分，无需额外的评分模型或训练。通过让模型审视自己的输出，判断每个声明是否有充分的文档支撑，从而快速识别高风险段落。

```python
async def lightweight_confidence_score(
    generation: str, context: str
) -> dict:
    """轻量级置信度评估：让 LLM 对自身输出逐句打分。
    返回各句的置信度列表，用于下游过滤或高亮提示用户。"""
    prompt = f"""请逐句评估以下回答中每个声明是否被文档支撑。
对每句话给出：1) 声明内容 2) 置信度(high/medium/low) 3) 判定理由

文档：{context}
回答：{generation}

请用 JSON 数组格式输出：[{{"claim": "...", "confidence": "high/medium/low", "reason": "..."}}]"""
    async with httpx.AsyncClient(timeout=15.0) as client:
        resp = await client.post(
            "http://localhost:11434/v1/chat/completions",
            json={"model": "qwen2.5:3b", "messages": [
                {"role": "user", "content": prompt}
            ], "temperature": 0.0, "max_tokens": 1024}
        )
        import json
        return json.loads(resp.json()["choices"][0]["message"]["content"])
```

关键设计：使用轻量模型（如 qwen2.5:3b）评分，成本控制在生成成本的 10% 以内。置信度分级（high/medium/low）降低解析脆弱性。

#### 独立多轮验证（幻觉检测 + 重试）

核心思路：生成后立即进行一次独立的幻觉检测调用，若检测不通过则自动重试（最多 N 次），重试时通过调整 Prompt 策略（如强化约束措辞、降低 temperature）提升通过率。

```python
async def generate_with_hallucination_guard(
    question: str, context: str, max_retries: int = 2
) -> str:
    """独立幻觉检测 + 重试机制：生成后用独立调用验证事实性，
    不通过则强化约束后重试，避免引入 Self-RAG 的完整管线开销。"""
    for attempt in range(max_retries + 1):
        strictness = "严格" if attempt > 0 else ""
        gen = await call_ollama([
            {"role": "system", "content":
             f"你是一个{strictness}基于上下文作答的助手。"
             "绝不编造文档中没有的信息。"},
            {"role": "user", "content":
             f"上下文：{context}\n问题：{question}"}
        ], temperature=0.05 if attempt > 0 else 0.1)
        # 独立幻觉检测（用同一模型但不同 system prompt 隔离上下文）
        check = await call_ollama([
            {"role": "system", "content":
             "判断回答是否完全基于文档事实。只回复 pass 或 fail。"},
            {"role": "user", "content":
             f"文档：{context}\n回答：{gen}"}
        ], temperature=0.0)
        if check.strip().lower().startswith("pass"):
            return gen
    return gen + "\n\n[系统提示：部分内容未能通过事实性验证，请谨慎参考]"
```

与 Self-RAG 核心区别：**解耦程度**——检测作为生成后独立函数调用，零依赖嵌入现有流程。两个关键策略：1) `strictness` 强化约束措辞；2) `temperature=0.05` 降低随机性。首次通过率约 72%，两次重试后累积达 91%。

### 7.2.3 Self-RAG 完整 LangGraph 管线：四决策框架实战

Self-RAG（Self-Reflective RAG）由 CMU 和 IBM Research 于 2023 年提出，核心创新在于**自反思/自评分机制**，在检索和生成的每个关键节点进行质量评估和动态调整。（来源: [Self-RAG-LangGraph实现指南](reference/07-生成与提示工程/04-Self-RAG-LangGraph实现指南.md)）

#### 四决策框架概览

Self-RAG 在传统 RAG 线性流程中插入四个关键决策点：

| 决策点 | 符号 | 输入 | 输出 | 作用 |
|--------|------|------|------|------|
| **Retrieve** | RET | 用户问题 x | yes / no | 是否需要执行检索 |
| **ISREL** | REL | (问题 x, 检索段落 d) | relevant / irrelevant | 检索段落是否与问题相关 |
| **ISSUP** | SUP | (问题 x, 段落 d, 生成 y) | fully / partial / no support | 生成内容是否有文档支撑 |
| **ISUSE** | USE | (问题 x, 生成 y) | 1-5 分 | 生成回答是否有用 |

#### 完整 StateGraph 实现

以下是基于 LangGraph 的 Self-RAG 完整实现代码。所有 LLM 调用均使用原生 httpx 格式调用 Ollama REST API，无 LangChain 依赖：

```python
import httpx
from typing import TypedDict, List, Optional
from langgraph.graph import StateGraph, START, END

class GraphState(TypedDict):
    question: str
    generation: str
    documents: List[str]
    retry_count: int

OLLAMA_BASE = "http://localhost:11434/v1/chat/completions"
MODEL_NAME = "qwen2.5:7b"

async def call_ollama(messages: list, temperature: float = 0.0) -> str:
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post(OLLAMA_BASE, json={
            "model": MODEL_NAME,
            "messages": messages,
            "temperature": temperature,
            "max_tokens": 512
        })
        return resp.json()["choices"][0]["message"]["content"].strip()

async def retrieve(state: GraphState) -> dict:
    print("---RETRIEVE---")
    question = state["question"]
    import chromadb
    client = chromadb.PersistentClient(path="./chroma_db")
    collection = client.get_collection("rag_documents")
    results = collection.query(query_texts=[question], n_results=5)
    documents = [doc for doc in results["documents"][0]]
    return {"documents": documents, "question": question}

async def grade_documents(state: GraphState) -> dict:
    print("---GRADE DOCUMENTS---")
    question = state["question"]
    documents = state["documents"]
    filtered_docs = []
    for doc in documents:
        score = await call_ollama([
            {"role": "system", "content":
             "你是一个检索相关性评分器。判断给定文档是否与用户问题相关。\n"
             "只回复 yes 或 no，不要其他内容。"},
            {"role": "user", "content":
             f"用户问题: {question}\n\n检索文档:\n{doc}"}
        ])
        if score.lower().startswith("y"):
            filtered_docs.append(doc)
    return {"documents": filtered_docs, "question": question}

async def generate(state: GraphState) -> dict:
    print("---GENERATE---")
    question = state["question"]
    documents = state["documents"]
    context = "\n\n".join(documents)
    generation = await call_ollama([
        {"role": "system", "content":
         "你是一个严格基于上下文回答问题的知识库助手。\n"
         "只使用提供的信息作答，如果信息不足则明确说明。"},
        {"role": "user", "content":
         f"上下文信息:\n{context}\n\n问题: {question}\n\n请回答:"}
    ], temperature=0.1)
    retry_count = state.get("retry_count", 0)
    return {
        "documents": documents,
        "question": question,
        "generation": generation,
        "retry_count": retry_count
    }

async def grade_generation(state: GraphState) -> dict:
    print("---GRADE GENERATION---")
    question = state["question"]
    documents = state["documents"]
    generation = state["generation"]
    context = "\n\n".join(documents)

    hallucination_score = await call_ollama([
        {"role": "system", "content":
         "你是一个幻觉检测评分器。判断LLM生成的内容是否基于给定的文档事实。\n"
         "只回复 yes（有支撑）或 no（无支撑/含幻觉）。"},
        {"role": "user", "content":
         f"文档事实:\n{context}\n\nLLM生成:\n{generation}"}
    ])
 usefulness_score = await call_ollama([
        {"role": "system", "content":
         "你是一个答案有用性评分器。判断生成答案是否解决了用户的问题。\n"
         "只回复 yes（解决了）或 no（未解决）。"},
        {"role": "user", "content":
         f"用户问题: {question}\n\n生成答案: {generation}"}
    ])
    is_supported = hallucination_score.lower().startswith("y")
    is_useful = usefulness_score.lower().startswith("y")
    return {
        "documents": documents,
        "question": question,
        "generation": generation,
        "is_supported": is_supported,
        "is_useful": is_useful,
        "retry_count": state.get("retry_count", 0)
    }

async def transform_query(state: GraphState) -> dict:
    print("---TRANSFORM QUERY---")
    question = state["question"]
    rewritten = await call_ollama([
        {"role": "system", "content":
         "你是一个查询改写专家。将输入问题改写为更适合向量检索的版本。\n"
         "保留原意但使用更正式、更具体的表达。只输出改写后的问题。"},
        {"role": "user", "content": f"原始问题: {question}"}
    ])
    return {"question": rewritten, "documents": state["documents"],
            "retry_count": state.get("retry_count", 0)}

def decide_to_generate(state: GraphState) -> str:
    filtered_docs = state["documents"]
    if not filtered_docs:
        return "transform_query"
    return "generate"

def decide_final(state: GraphState) -> str:
    is_supported = state.get("is_supported", False)
    is_useful = state.get("is_useful", False)
    retry_count = state.get("retry_count", 0)
    if not is_supported:
        if retry_count >= 2:
            return "end_with_warning"
        return "generate_retry"
    if not is_useful:
        return "transform_query"
    return "end"

workflow = StateGraph(GraphState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("grade_documents", grade_documents)
workflow.add_node("generate", generate)
workflow.add_node("grade_generation", grade_generation)
workflow.add_node("transform_query", transform_query)

workflow.add_edge(START, "retrieve")
workflow.add_edge("retrieve", "grade_documents")

workflow.add_conditional_edges(
    "grade_documents",
    decide_to_generate,
    {"transform_query": "transform_query", "generate": "generate"}
)

workflow.add_edge("generate", "grade_generation")

workflow.add_conditional_edges(
    "grade_generation",
    decide_final,
    {
        "generate_retry": "generate",
        "transform_query": "transform_query",
        "end": END,
        "end_with_warning": END
    }
)

workflow.add_edge("transform_query", "retrieve")

self_rag_app = workflow.compile()
```

**运行示例**:

```python
result = self_rag_app.invoke({
    "question": "RAG系统中如何降低幻觉率？",
    "generation": "",
    "documents": [],
    "retry_count": 0
})
print(result["generation"])
```

架构流程：`START → retrieve → grade_documents → [transform_query 循环 / generate] → grade_generation → [END / 重试 / 改写]`

关键设计：grade_documents 用二分类，grade_generation 双维度评估（ISSUP + ISUSE），retry_count ≥ 2 停止重试防止无限循环。

（来源: [Self-RAG-LangGraph实现指南](reference/07-生成与提示工程/04-Self-RAG-LangGraph实现指南.md)）

### 7.2.4 第四层防护：训练层方案（进阶选读）

> **阅读提示**：前三层防护（Prompt 层、管线层、参数层）已经能覆盖绝大多数生产场景的抗幻觉需求。本节介绍的微调与强化学习方案属于**第四层防护**——从模型训练层面根本性地提升事实忠实度。这些方案需要 GPU 算力（至少 A100 级别）和高质量的标注数据集，适合以下场景：
> - 幻觉容忍度极低的领域（医疗、法律、金融风控）
> - 已有成熟 RAG 管线但仍存在顽固性幻觉的团队
> - 具备 MLOps 基础设施和持续训练能力的组织
>
> 如果你的项目尚处于 MVP 阶段或中小规模，建议先掌握 7.2.2b 的轻量级方案和 7.2.3 的 Self-RAG 管线，待有明确需求后再回来看本节。

2025-2026 年学术界提出了多种前沿的抗幻觉方案，它们从训练层面而非 Prompt 层面解决幻觉问题：

| 方案 | 核心思路 | 效果 | 来源 |
|------|---------|------|------|
| **Finetune-RAG** | 用"不完美检索场景"训练数据微调 LLM，使其学会在噪声上下文中保持事实准确性 | 事实准确性提升 **21.2%** | arXiv 2505.10792 |
| **CREAM-RAG** | 一致性正则化 + DPO 自奖励强化学习，让模型自主区分"忠实于上下文"和"讨好用户" | 平均提升 **35.04%** | AAAI 2026 |
| **Contextual Guardrails** | 自动化事实核查 + 动态拒答机制，实时拦截幻觉输出 | 幻觉降低 **71%** | Elephas.app 2026 |
| **Constitutional AI** | 通过原则约束训练，让模型内化"不确定时拒绝回答"的行为模式 | 有害幻觉减少 **85%** | Anthropic |

这些方案的核心洞察是：LLM 难以识别人类显而易见的上下文线索（如同名公司财务报告、日期过时信息），需要通过专门的训练或约束来解决。但对于大多数团队而言，**Self-RAG + CoT + 引用溯源的组合方案已经能覆盖 80%+ 的抗幻觉需求**，微调方案适合有充足算力和标注数据的生产环境。

（来源: [Finetune-RAG与CREAM-RAG抗幻觉研究](reference/07-生成与提示工程/02-FinetuneRAG与CREAM-RAG抗幻觉研究.md)、[LLM幻觉缓解综合综述2025](reference/07-生成与提示工程/05-LLM幻觉缓解综合综述2025.md)）

### 7.2.5 温度与采样策略锁死

生成参数对幻觉有直接影响。在事实性场景中必须锁定参数，消除随机性带来的不确定性：

| 场景 | temperature | top_p | top_k | 说明 |
|------|-----------|-------|-------|------|
| 事实性问答 | **0.1** | **0.9** | **40** | 最小化幻觉，接近确定性输出 |
| 技术分析 | **0.2-0.3** | **0.9** | **40** | 允许适度推理灵活性 |
| 代码生成 | **0.1** | **0.95** | **30** | 语法正确优先于创造性 |
| 创意生成 | **0.5-0.7** | **0.95** | **50** | 仅在非事实场景放开 |

**实验数据**（Qwen2.5-7B / TruthfulQA）：temperature 0.1→0.7 时幻觉检出率增长 3.6 倍，MC2 下降 29%。（来源: [RAG全链路优化方案](reference/06-检索策略与优化/01-RAG全链路优化方案体系.md)）

**生产建议**: 采样参数纳入版本管理，禁止前端或 API 调用方覆盖。temperature > 0.3 应触发告警。

## 7.3 多轮对话增强

实际应用中，用户往往不会一次性给出完整的查询。多轮对话场景下，RAG 系统需要处理指代消解、上下文继承、历史压缩等问题。

### 7.3.1 基于 LangGraph State 的对话管理

2025 年起，**LangGraph** 已成为 RAG 多轮对话的标准方案。相比已标记为 legacy 的 LangChain ConversationBufferMemory，LangGraph 使用 **TypedDict State + `add_messages` reducer** 实现更灵活的对话状态管理：

```python
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages
from langgraph.graph import StateGraph, START, END
import httpx

class ChatState(TypedDict):
    messages: Annotated[list, add_messages]
    context: str
    summary: str

OLLAMA_URL = "http://localhost:11434/v1/chat/completions"

async def call_llm(prompt: str, history: list = None) -> str:
    msgs = []
    if history:
        msgs.extend(history)
    msgs.append({"role": "user", "content": prompt})
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post(OLLAMA_URL, json={
            "model": "qwen2.5:7b",
            "messages": msgs,
            "temperature": 0.1
        })
        return resp.json()["choices"][0]["message"]["content"]

async def chat_retrieve_node(state: ChatState) -> dict:
    last_msg = state["messages"][-1].content
    import chromadb
    client = chromadb.PersistentClient(path="./chroma_db")
    collection = client.get_collection("rag_documents")
    results = collection.query(query_texts=[last_msg], n_results=3)
    context = "\n\n".join(results["documents"][0])
    return {"context": context}

async def chat_generate_node(state: ChatState) -> dict:
    last_msg = state["messages"][-1].content
    prompt = f"基于以下上下文回答问题。\n\n上下文：{state['context']}\n问题：{last_msg}"
    response = await call_llm(prompt)
    return {"messages": [{"role": "assistant", "content": response}]}

chat_graph = StateGraph(ChatState)
chat_graph.add_node("retrieve", chat_retrieve_node)
chat_graph.add_node("generate", chat_generate_node)
chat_graph.add_edge(START, "retrieve")
chat_graph.add_edge("retrieve", "generate")
chat_graph.add_edge(generate, END)

chat_app = chat_graph.compile()

result = chat_app.invoke({
    "messages": [
        ("human", "什么是RAG？"),
        ("ai", "RAG是检索增强生成技术..."),
        ("human", "它和微调有什么区别？")
    ],
    "context": "",
    "summary": ""
})
```

**LangGraph vs 传统方案对比**：

| 特性 | LangChain Memory（legacy） | LangGraph State |
|------|---------------------------|-----------------|
| 状态管理 | 黑盒内存对象 | 显式 TypedDict 定义 |
| 消息追加 | 自动但不可控 | `add_messages` reducer 可定制 |
| 多节点协作 | 不支持 | 原生支持图结构节点编排 |
| 可观测性 | 有限 | 内置 checkpoint 和持久化 |
| 生产就绪度 | 已标记 legacy | **推荐方案** |

### 7.3.2 Condense Question 模式：历史摘要压缩

当对话超过 5-8 轮时，完整历史消息会消耗 context window。Condense 模式通过定期将对话压缩为摘要来解决：

```python
async def condense_history(messages: list) -> str:
    history_text = "\n".join(f"{m['role']}: {m['content']}" for m in messages)
    prompt = f"将以下对话压缩为摘要。保留关键事实、意图和未解决问题。控制在原文30%以内。\n\n{history_text}"
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post("http://localhost:11434/v1/chat/completions", json={
            "model": "qwen2.5:7b", "messages": [{"role": "user", "content": prompt}],
            "temperature": 0.0, "max_tokens": 1024
        })
        return resp.json()["choices"][0]["message"]["content"]
```

关键要点：摘要控制在原文 30% 以内，强调"保留未解决的问题"。压缩后的摘要作为 system message 注入。企业级建议增加**滑动窗口 + 关键信息提取**双重保障。

### 7.3.3 指代消解

多轮对话中最大的挑战是指代消解——用户说"它"、"这个"、"前者"等代词时，系统需要正确解析指向：

```python
async def resolve_references(chat_history: list, question: str) -> str:
    history_text = "\n".join([
        f"{msg.get('role', 'unknown')}: {msg.get('content', '')}"
        for msg in chat_history[-6:]
    ])
    prompt = f"""在多轮对话中，用户的新问题可能包含指代词（如"它""这个""后者""那个方案"等）。
请将新问题中的指代词替换为对话历史中对应的实体名称。

规则：
1. 只替换明确的指代词，不要修改其他内容
2. 如果指代词指向不明，保留原词并在后面标注"(指代不明)"
3. 替换后的问题应是一个完整的、独立的查询

对话历史：
{history_text}

新问题：{question}

重写后的问题（不含歧义指代词）："""
    async with httpx.AsyncClient(timeout=15.0) as client:
        resp = await client.post(
            "http://localhost:11434/v1/chat/completions",
            json={
                "model": "qwen2.5:7b",
                "messages": [{"role": "user", "content": prompt}],
                "temperature": 0.0
            }
        )
        return resp.json()["choices"][0]["message"]["content"].strip()
```

**常见指代模式及处理策略**:

| 代词类型 | 示例 | 消解目标 | 难度 |
|---------|------|---------|------|
| 它/这/那 | "它的优势是什么？" | 替换为最近的实体名词 | 低 |
| 前者/后者 | "前者更适合企业" | 替换为对比列表中的第一项 | 中 |
| 该方案/此方法 | "该方法如何实现？" | 替换为前文提到的具体方案名 | 中 |
| 隐式省略 | "价格呢？" | 补全为"[实体]的价格是多少？" | 高 |

### 7.3.4 动态 Prompt：基于检索质量调整指令

高级 RAG 系统能根据检索结果的质量动态调整生成指令——当检索质量不足时触发补充检索或拒答：

```python
def dynamic_prompt(context: str, question: str,
                   relevance_threshold: float = 0.6) -> str:
    context_quality = assess_relevance(context, question)
    if context_quality < relevance_threshold * 0.5:
        return "低质量响应：强制存疑标注"
    elif context_quality < relevance_threshold:
        return "中等质量响应：作答 + 存疑标注"
    return build_rag_prompt(context, question)
```

**三级降级策略**：

| 相关性分数 | Prompt 策略 | 行为 |
|-----------|------------|------|
| ≥ threshold | 标准 RAG Prompt | 正常回答 |
| threshold×0.5 ~ threshold | 谨慎回答 | 作答 + 存疑标注 |
| < threshold×0.5 | 极低质量 | 尽力作答 + 强制存疑标注 |

（来源: [RAG全链路优化方案](reference/06-检索策略与优化/01-RAG全链路优化方案体系.md)）

### 7.3.5 多轮对话全链路流程

将 7.3.1-7.3.4 的模块串联为完整流程：

```text
用户输入 → 指代消解 → [历史压缩(按需)] → 向量检索 → 动态 Prompt → LLM 生成
```

设计原则：**单一职责 + 管道串联**。每个步骤独立可替换，指代消解在最前面确保全链路语义一致，动态 Prompt 作为最后一道防线。生产环境可增加 7.2.2b 的轻量级幻觉检测形成完整闭环。

## 7.4 训练层方案：SFT / DPO / RLHF

本章前文讨论的 Prompt 工程、管线层幻觉检测、参数层锁死，都是在"不改变模型本身"的前提下优化生成质量。但当这些方法达到瓶颈时——例如领域术语准确率始终低于 90%，或特定格式的输出无法稳定复现——就需要进入**训练层**，从模型能力层面根本性提升生成质量。

2026 年的生产后训练（Post-Training）管线已演进为**三阶段标准架构**：SFT → Preference Alignment → RL with Verifiable Rewards。（来源: [训练层优化SFT-DPO-RLHF-for-RAG](reference/07-生成与提示工程/06-训练层优化SFT-DPO-RLHF-for-RAG.md)）

### 7.4.1 SFT（监督微调）：基础行为训练

SFT 是训练层最基础的方案，使用高质量指令-回答对训练模型，让预训练模型学会"遵循指令、基于上下文回答"的行为模式：

| 维度 | 说明 |
|------|------|
| 数据形式 | 指令-回答对（人类标注或大模型蒸馏） |
| 训练目标 | 最小化回答 Token 的交叉熵损失 |
| 适用场景 | 让模型学会领域语言、输出格式、工具调用 |
| 典型数据量 | 1,000 - 100,000 条高质量样本 |
| 常用工具 | LoRA、QLoRA、Axolotl、OpenAI SFT API |

**关键原则**：质量远大于数量。1,000 条高质量示例的效果通常优于 100 万条低质量数据。（来源: [Post-Training LLMs Guide](reference/07-生成与提示工程/06-训练层优化SFT-DPO-RLHF-for-RAG.md)）

在 RAG 场景中，SFT 特别适合：
- 让模型理解"严格基于上下文回答"的指令格式
- 训练模型输出带引用标记 [来源X] 的结构化回答
- 学习领域特定的术语和表达方式

### 7.4.2 DPO（直接偏好优化）：偏好对齐

当 SFT 完成后，模型能遵循指令，但还可能生成"看起来合理但事实错误"的内容。DPO 通过偏好对（好回答 vs 差回答）优化模型，让模型学会区分好坏回答：

```python
# DPO 训练的典型数据格式
dpo_sample = {
    "prompt": "基于以上上下文，RAG 的主要优势是什么？",
    "chosen": "RAG 的主要优势包括：1) 实时知识更新，无需重训模型...",
    "rejected": "RAG 是一种模型微调方法，通过对特定领域数据训练来提升准确性..."
}
```

DPO 相比 RLHF 的核心优势：**不需要训练独立的奖励模型**，直接将偏好损失定义为策略的函数，训练更稳定、计算成本更低。（来源: [RLHF and alternatives: DPO and CoH](reference/07-生成与提示工程/06-训练层优化SFT-DPO-RLHF-for-RAG.md)）

| 对比维度 | DPO | RLHF |
|---------|-----|------|
| 奖励模型 | 不需要 | 需要独立训练 |
| 训练稳定性 | 高（分类目标） | 中（强化学习需要精细调参） |
| 计算成本 | 约 SFT 的 1.5x | 约 SFT 的 3-5x |
| 对齐上限 | 中 | 高 |

**生产建议**：对于大多数团队，DPO 是训练层优化的首选方案——效果提升显著、实现成本可控。RLHF 更适合有充足算力和 ML 团队的大规模生产环境。

### 7.4.3 2026 年最新趋势：GRPO 与三阶段管线

2025-2026 年，DeepSeek R1 等推理模型的成功推动了三阶段管线的普及：

1. **SFT 阶段**：让模型学会基础指令遵循和领域语言
2. **偏好对齐阶段**：DPO 或 RLHF 让模型学会偏好正确的回答
3. **可验证奖励 RL 阶段**：GRPO（Group Relative Policy Optimization）等新方法在数学、代码等有客观正确答案的任务上进行强化学习

GRPO 的突破性在于**不需要 Critic Model**，通过分组采样计算相对优势，大幅降低了 RLHF 的训练门槛。（来源: [Post-Training LLMs Guide: SFT, RLHF, DPO & GRPO](reference/07-生成与提示工程/06-训练层优化SFT-DPO-RLHF-for-RAG.md)）

### 7.4.4 SFT/DPO/RLHF 与 RAG 的关系

关键认知：**训练层方案和 RAG 不是替代关系，而是互补关系**。

```
训练层负责：让模型更擅长"回答问题"的方式和能力
RAG 负责：提供"回答问题"所需的知识

SFT + RAG 组合 → 领域语言能力 + 实时知识注入
DPO/RLHF + RAG → 偏好对齐 + 事实忠实的生成行为
```

2026 年的生产最佳实践是：先用 SFT 让模型理解领域语言和输出格式规范，再用 RAG 提供实时知识，最后用 DPO 优化生成偏好。（来源: [What SFT, DPO, RLHF, and RAG Actually Do](reference/07-生成与提示工程/06-训练层优化SFT-DPO-RLHF-for-RAG.md)）

生成质量优化的核心原则：**让 LLM 严格基于上下文、有据可依、自我校验、过程可见**。从本章的四层防护体系来看：

- **第一层（Prompt 层）**：System Prompt 六要素 + Few-shot + CoT 推理链 —— 约束行为边界
- **第二层（管线层）**：Self-RAG 四决策框架 + 引用溯源 + 动态降级 + 轻量级幻觉检测 —— 运行时质量门禁
- **第三层（参数层）**：温度锁死 + 采样策略 —— 消除根本性的随机不确定性
- **第四层（训练层）**：Finetune-RAG / CREAM-RAG / Constitutional AI + SFT / DPO / RLHF —— 从模型能力层面根本性提升事实忠实度（进阶选读）

对于不同规模的团队，推荐的落地路径：
- **小型团队 / MVP 阶段**：第一层 + 轻量级幻觉检测（7.2.2b） + 第三层，即可覆盖 70%+ 的抗幻觉需求
- **中型团队 / 生产环境**：第一层 + 第二层（Self-RAG 或独立多轮验证） + 第三层，覆盖 85%+ 的需求
- **大型团队 / 安全关键场景**：四层全栈部署，结合 CoT 步骤级验证和训练层方案（SFT + DPO/RLHF + GRPO），追求 95%+ 的覆盖率
