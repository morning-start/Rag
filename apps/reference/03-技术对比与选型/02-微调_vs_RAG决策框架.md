## 核心结论

**Fine-Tuning**: 更新模型权重 — 改变模型**如何思考**
**RAG**: 保持模型冻结，在推理时检索实时文档 — 改变模型**看到什么**

当不确定时，**从 RAG 开始**。

## 六维决策框架

| 决策因素 | Fine-Tuning | RAG |
|----------|-------------|-----|
| **数据更新频率** | 知识固化在训练时，变更后立即过时 | **动态** — 从实时知识库检索，立即反映更新 |
| **治理成熟度** | 不良训练数据永久产生不良输出 | 治理良好：85-92%准确率；治理不良：45-60% |
| **延迟** | **更低** — 无检索步骤 | 较高 — 检索增加 50-200ms |
| **成本** | 高 — 每次训练运行 $50K-$500K | **低** — 更新知识库，无需重新训练 |
| **可解释性** | 低 — 黑箱生成 | **高** — 可追踪来源 |
| **定制化程度** | **高** — 适合领域术语与逻辑 | 中等 — 依赖检索质量 |

## 适用场景

### Fine-Tuning 适用
- 固定领域任务（如法律文书、医疗诊断）
- 需要改变模型行为、输出格式、语气风格
- 知识稳定，变更频率低于季度
- 延迟敏感（< 200ms）
- 高查询量场景

### RAG 适用
- 需要实时/最新信息
- 知识频繁更新（月度或更快）
- 需要来源追溯和引用
- 数据隐私敏感（私有数据本地部署）
- 中等延迟可接受（200ms - 2s）

## 融合方案（推荐最佳实践）

### 两阶段优化
1. 先通过 Fine-Tuning 让模型掌握领域基础逻辑
2. 再引入 RAG 补充实时知识

### 动态混合框架
- 根据任务复杂度自动切换模式
- 简单问题用 Fine-Tuning
- 复杂检索启用 RAG

### 示例场景
**医疗助手**:
- Fine-Tune 用于临床推理和术语理解
- RAG 用于药物数据库、最新诊疗指南

**客服机器人**:
- Fine-Tune 用于统一服务话术和语气
- RAG 用于产品信息、订单状态等实时数据

## 总体建议

| 场景 | 推荐方案 |
|------|----------|
| 快速验证、简单任务 | 提示工程 |
| 需要外部知识、知识需频繁更新 | **RAG** |
| 需要改变模型行为、特定任务优化 | **Fine-Tuning** |
| 生产级企业应用 | **RAG + Fine-Tuning** |
