## 一、为什么分块对 RAG 至关重要

文档分块是 RAG 管线的**第一步**，也是最容易出问题的一步。分块质量直接决定了：

1. **LLM Context Window 限制**: 大模型有固定的上下文窗口大小，必须将长文本切分为可处理的片段
2. **信噪比问题**: chunk 过大 → 核心信号被噪声稀释；chunk 过小 → 语义不完整，无法支撑准确回答

**核心平衡点**: 每个 chunk 必须足够小以保持聚焦，又必须足够大以保留完整语义。

## 三、各策略详解与代码示例

### 3.1 Fixed-Size Chunking（固定大小分块）

**原理**: 按固定字符数或 token 数机械切割，不考虑语义边界。

```python
from langchain_text_splitters import CharacterTextSplitter

text_splitter = CharacterTextSplitter(
    separator=" ",
    chunk_size=512,
    chunk_overlap=50,
    length_function=len,
)
docs = text_splitter.create_documents([document])
```

- **优点**: 实现简单、速度最快、chunk 大小可预测
- **缺点**: 可能从句子中间切断，破坏语义完整性
- **适用**: 日志文件、表格化文本、均匀记录数据；仅用于原型验证

### 3.2 Recursive Character Chunking（递归字符分块）

**原理**: 按优先级分隔符列表 `["\n\n", "\n", " ", ""]` 层级式分割。先尝试段落边界，再尝试句子、单词。

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=50,
)
docs = text_splitter.create_documents([document])
```

- **优点**: 尊重文档多层结构、产生更连贯的 chunk、LangChain 默认推荐
- **缺点**: 分隔符层级可能不完全匹配所有文档格式
- **关键 Benchmark**: Vecta 2026 年 2 月在 50 篇学术论文上的测试中，**Recursive 512-token 分块达到 69% 端到端准确率**，超过更复杂的策略（来源: AtlasSC）
- **适用**: **绝大多数 RAG 应用的默认起点**

### 3.3 Token-Based Chunking（基于 Token 的精确分块）

**原理**: 使用与 embedding 模型/LLM 相同的 tokenizer 进行切分，确保精确控制 token 数量。

```python
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

def chunk_by_tokens(text, chunk_size=512, overlap=64):
    tokens = enc.encode(text)
    chunks = []
    start = 0
    while start < len(tokens):
        end = min(start + chunk_size, len(tokens))
        chunks.append(enc.decode(tokens[start:end]))
        if end == len(tokens):
            break
        start = max(0, end - overlap)
    return chunks
```

- **优点**: 与模型 token 限制精确对齐、适合多语言场景
- **适用**: 严格的 context window / API 成本限制场景

### 3.4 Sentence-Based Chunking（句子级分块）

**原理**: 以句子为原子单位进行分组，保证不会从句子中间切断。

```python
import nltk
from nltk.tokenize import sent_tokenize

def chunk_by_sentences(text, max_chars=500, overlap_sentences=1):
    sentences = sent_tokenize(text)
    chunks = []
    current_chunk = ""
    for i, sentence in enumerate(sentences):
        if len(current_chunk) + len(sentence) <= max_chars:
            current_chunk += " " + sentence
        else:
            chunks.append(current_chunk.strip())
            start_index = max(0, i - overlap_sentences)
            current_chunk = " ".join(sentences[start_index:i+1])
    if current_chunk:
        chunks.append(current_chunk.strip())
    return chunks
```

- **注意**: NLTK 等标准库对中文支持有限（中文句号 `。`），中文场景需使用语言特定模型或正则
- **适用**: 法律文书、新闻报道等要求句子完整性的场景

### 3.5 Document-Based / Structure-Aware Chunking（结构感知分块）

**原理**: 利用文档自身的格式标记（Markdown 标题、HTML 标签、代码块边界）定义 chunk 边界。

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

headers_to_split_on = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]
splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
md_header_splits = splitter.split_text(markdown_document)
```

- **优点**: 每个 chunk 天然对应一个完整主题/章节，元数据可保留标题层级
- **缺点**: 依赖文档有良好的结构标记；纯文本/OCR 输出不适用；chunk 大小可能极不均匀
- **适用**: 技术文档（Markdown）、网页内容（HTML）、代码仓库

### 3.6 Semantic Chunking（语义分块）

**原理**: 计算相邻句子的向量相似度（cosine distance），当相似度显著下降时插入分块边界——表示主题发生了变化。

```python
from langchain_experimental.text_splitter import SemanticChunker
from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")
text_splitter = SemanticChunker(
    embeddings,
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=70
)
chunks = text_splitter.create_documents([document])
```

- **Benchmark 数据**: Semantic chunking 在基准测试中相比朴素 baseline 可提升约 **70% 准确率**（来源: LangCopilot）
- **优点**: 每个 chunk 代表连贯的主题思想，不受格式标记约束
- **缺点**: 每个句子都需要一次 embedding 调用，增加延迟和成本；chunk 大小不可预测（Vecta 2026 benchmark 发现 semantic chunking 平均仅 43 tokens，检索干净但给 LLM 的上下文过少）
- **NAACL 2025 反面发现**: 固定 200-word chunk 在检索和生成任务上**匹配或优于**语义分块，说明计算开销并不总是值得
- **适用**: 多主题交错的研究论文、会议纪要、客服对话线程

### 3.7 LLM-Based Chunking（LLM 引导分块）

**原理**: 使用 LLM 阅读全文并识别语义完整的独立命题（proposition），每个命题作为独立 chunk。

```python
from langchain.chains import create_extraction_chain

schema = {
    "properties": {
        "proposition": {
            "type": "string",
            "description": "A standalone, semantically complete statement"
        }
    },
    "required": ["proposition"]
}
chain = create_extraction_chain(schema, llm)
propositions = chain.run(document)
```

- **优点**: 最高质量的语义完整性；LLM 可解析代词指代（如将"它"替换为实际实体名）
- **缺点**: **最昂贵的策略**——每篇文档都需 LLM 处理；不适合大规模实时摄入
- **适用**: 法律文档分析、医疗知识库、金融合规语料等**错误代价极高**的领域

### 3.8 Late Chunking（延迟分块 / 上下文化分块嵌入）

**原理**: Jina AI 于 2024 年提出。颠覆传统"先分块再嵌入"流程：先将**整篇文档**送入长上下文 embedding 模型生成 token 级嵌入，再应用分块边界，最后对每个 chunk 内的 token 嵌入做 pooling。

```python
import requests

response = requests.post(
    "https://api.jina.ai/v1/embeddings",
    headers={"Authorization": "Bearer YOUR_API_KEY"},
    json={
        "model": "jina-embeddings-v3",
        "input": ["First chunk boundary text.", "Second chunk boundary text."],
        "late_chunking": True
    }
)
```

- **核心优势**: 每个 chunk 的嵌入手持整篇文档的上下文信息——chunk 5 中的"他们"会带着 chunk 1 中"他们指代谁"的知识被编码
- **限制条件**: 输入文档必须能放入 embedding 模型的 context window（当前主流 8K-32K tokens）；依赖暴露 token 级表示的特定模型架构
- **适用**: 跨章节的长叙事文档：技术手册、法律合同、研究论文、需要代词消解和交叉引用理解的文档

### 3.9 Agentic Chunking（智能体分块）

使用 AI Agent 动态决定分块边界，适用于复杂多格式文档的自动处理。（详细实现见 LangCopilot 原文）

## 四、实践默认值（2026 年 2 月重新验证）

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **chunk_size** | **256-512 tokens** | 大多数场景的最佳平衡点 |
| **chunk_overlap** | **10-20%**（50-100 tokens for 512-token chunks） | 防止关键信息在边界处被切断 |
| **最佳准确率策略** | Semantic chunking | ~70% 检索质量提升 |
| **最佳速度策略** | Recursive chunking | LangChain 默认，零额外开销 |
| **最佳成本策略** | Token-based with caching | 精确控制 API 调用量 |

## 五、2026 年升级方案：超越调参的新范式

### 5.1 Contextual Retrieval（上下文化检索）

Anthropic 于 2024 年 9 月提出。在每个 chunk 被 embedding **之前**，用 LLM 为其生成一段来自整篇文档的上下文描述并**前置拼接**到 chunk 内容上。

- 效果: 显著改善 chunk 的语义独立性和检索相关性
- 成本: 每个 chunk 需要 1 次 LLM 调用
- 适用: 对检索精度要求极高的场景

### 5.2 Cross-Granularity Retrieval（跨粒度检索）

- 将同一内容存储在**多个不同粒度级别**（sentence-level + paragraph-level + document-level）
- 查询时根据查询类型动态选择合适的粒度进行检索
- 类似 ParentDocumentRetriever 思想的系统化扩展

### 5.3 Hybrid Chunking（混合分块）

组合多种策略应对复杂文档：
- Markdown/HTML 部分 → 结构感知分块
- 代码部分 → 函数/类级别分块
- 叙述性文本 → 语义分块或递归分块

## 六、决策指南速查表

| 你的需求 | 推荐策略 | 说明 |
|----------|----------|------|
| **最高准确率** | Semantic chunking | ~70% 提升，但需监控 chunk 大小分布 |
| **通用 RAG** | **Recursive 512-token** | 默认首选，69% 端到端准确率 |
| **预算紧张 / API 限额** | Token-based chunking | 精确 token 控制，可配合缓存 |
| **刚入门 / 快速原型** | RecursiveCharacterTextSplitter | LangChain 默认，开箱即用 |
| **结构化文档（MD/HTML/代码）** | Document-Based + Recursive fallback | 最自然的逻辑分块 |
| **检索到无关内容但 recall 好** | Semantic chunking + 设置最小 size 阈值 | 解决主题漂移问题 |
| **高风险领域（法律/医疗/金融）** | LLM-Based chunking | 成本高但质量最高 |
| **跨 chunk 上下文丢失严重** | **Late Chunking** | 保留全局文档上下文 |
| **复杂多格式文档** | Hybrid Chunking | 按内容类型选用不同策略 |

## 七、重要反直觉发现

### Vectara NAACL 2025 研究

> **固定分块在真实文档集上端到端优于语义分块**

- 在真实文档集上测量端到端效果（而非仅检索指标），固定大小的 chunk 表现优于语义分块
- 识别出 **"context cliff"（上下文悬崖）** 现象：响应质量在约 **~2500 tokens** 处出现显著下降
- **启示**: 始终用你的实际数据进行端到端 benchmark，不要仅依赖检索 recall 指标

### Semantic Chunking 的"小片段陷阱"

- Vecta 2026 benchmark: semantic chunking 产生的片段平均仅 **43 tokens**
- 检索很精准，但给 LLM 的上下文太少，导致生成答案不正确
- **对策**: 设置最小 chunk size 阈值（如 ≥100 tokens）

| 企业知识库（技术文档为主） | Recursive + Document-Based fallback | Semantic |

## 关键要点速记

1. **从 Recursive 512-token 开始**——这是经过最多生产验证的基线
2. **overlap 设为 10-20%**——这是防止信息丢失的安全网
3. **Semantic chunking 提升大但成本也大**——仅在确实需要时启用
4. **始终端到端评估**——检索 recall ≠ 最终答案质量
5. **context cliff 在 ~2500 tokens**——超长 context 不一定更好
6. **Late Chunking 是 2025-2026 的热点方向**——特别适合长文档场景
7. **没有万能最优策略**——最优选择取决于你的文档类型、查询模式和预算
